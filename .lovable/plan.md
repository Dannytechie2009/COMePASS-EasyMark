
# JAMB CBT Tutorial Platform — Build Plan

## Stack note (important)
You chose Firebase + Cloudinary. The Lovable app runs on TanStack Start (React 19) deployed to Cloudflare Workers. That means:
- **Firebase Auth + Firestore** will be used via the **Firebase Web SDK** directly from the browser (works perfectly). Security rules in Firestore will enforce access — not server middleware.
- **Cloudinary** will use **unsigned upload presets** for image uploads from the browser (standard pattern; no server needed).
- No Firebase Admin SDK on the server (it's Node-only and won't run on Workers). Anything an "admin" does is enforced via Firestore Security Rules + custom claims.
- Trade-off vs Lovable Cloud: you'll manage Firebase/Cloudinary accounts and API keys yourself, and email verification/password reset use Firebase's default emails (not your domain) unless you configure SMTP in Firebase.

If at any point you want to switch to Lovable Cloud, the data model below ports cleanly.

## Roles
- **student** — registers, takes exams
- **tutor** — creates/manages questions and exam sessions
- **super_admin** — everything tutor can do + manage users, generate product keys, reset passwords

Roles stored as Firebase Auth **custom claims** + a `users/{uid}` Firestore doc. Custom claims are set by a tiny Cloud Function (one-time setup) or seeded manually in Firebase console for the first super admin.

## Data model (Firestore)

```
users/{uid}
  name, email, role, department, subjects[4], createdAt, emailVerified

studentIds/{idNumber}        # human-friendly ID shown to student after registration
  uid, createdAt

productKeys/{key}            # generated by super admin
  uid, examType: 'combo'|'single', trialsTotal, trialsUsed, expiresAt, redeemed

questions/{qid}
  subject, text, options[4], correctIndex, explanation?, imageUrl?, createdBy, createdAt

examSessions/{sessionId}     # an exam instance scheduled by a tutor
  title, mode: 'single'|'combo', subjects[], questionIds[] (per subject),
  durationMinutes, startAt, requiresProductKey: bool,
  shuffleQuestions: bool, showCorrectionsAt: timestamp|null,
  createdBy, createdAt, status: 'scheduled'|'live'|'ended'|'corrections_open'

attempts/{sessionId}_{uid}   # one per student per session (enforces "one attempt")
  startedAt, endedAt, answers: { [qid]: chosenIndex },
  questionOrder: [qid],      # the per-student shuffled order, persisted
  score, totalPossible, autoSubmitted: bool, productKeyUsed?
```

**Critical rules:**
- `attempts/{sessionId}_{uid}` doc ID guarantees one attempt per student per session.
- `questionOrder` is persisted on first load so refresh keeps the same numbering and saved answers.
- Server time (Firestore `serverTimestamp`) is used for `startedAt`; deadline = `startedAt + durationMinutes`. Refresh re-reads this — timer keeps counting even after refresh.
- During exam: students can only read questions referenced in their `questionOrder`, and only `text` + `options` (NOT `correctIndex`). Enforced in Firestore Security Rules.
- After tutor flips `status` to `corrections_open`: students can read full questions including `correctIndex` for that session.

## Feature breakdown

### 1. Auth & accounts
- Email/password signup with Firebase email verification
- Password reset via Firebase
- Registration form captures: name, department (Science/Commercial/Art), subject combination (validated)
- On registration, generate a short human-readable **student ID** (e.g. `JMB-7F3K2X`) and show it to the student
- Login routes by role: student → student dashboard, tutor/super_admin → admin dashboard

### 2. Subject combination validator
Single source of truth in `src/lib/subjects.ts`:
- Science: English + Physics + Chemistry + (Biology XOR Mathematics)
- Commercial: English + Mathematics + pick 2 of {Commerce, Economics, Financial Accounting}
- Art: English + Government + Literature + CRS

UI enforces this with department-aware checkboxes; impossible combos are disabled.

### 3. Question bank (admin)
- Tutor creates/edits/deletes questions per subject
- Fields: text, 4 options, correct answer, optional explanation, optional image (Cloudinary upload)
- Filter by subject; bulk import (CSV) deferred to v2

### 4. Single-subject scheduled exam
- Tutor picks subject, selects N questions (or "random N from bank"), sets duration + start time, optional `requiresProductKey`
- Student dashboard shows upcoming/live exams for their subjects
- Lobby page counts down to `startAt`; "Start" enabled at start time
- Exam UI: one question at a time or scrollable list (configurable), timer top-right, "Submit" button
- Auto-submit on timeout
- After submit: shows **score only**
- Tutor "Release corrections" button → flips status; students can now see questions + their answers + correct answer in the original (non-shuffled) order

### 5. Full 4-subject combination exam
- Tutor creates a combo session by picking 4 existing subject question sets
- Always `requiresProductKey: true`
- Each subject has its own duration (or one shared); standard JAMB format defaults available
- Student flow:
  1. Student visits combo exam, prompted for product key
  2. Key validated: matches their uid, not expired, `trialsUsed < trialsTotal`
  3. On start, `trialsUsed` increments; attempt doc created
- Same anti-refresh/single-attempt-per-trial protections

### 6. Product key generation (super admin)
- Super admin page: search student by ID or name → enter trials count → generate key
- Key format: `XXXX-XXXX-XXXX` (cryptographically random, stored in `productKeys`)
- Shown once to super admin to hand off; student enters it in their exam lobby

### 7. Live scoreboard
- Per session, real-time Firestore subscription to `attempts` where `sessionId == X`
- Shows: rank, name, score, time taken, status (in-progress / submitted)
- Visible to tutors during/after; configurable visibility to students after corrections release

### 8. Admin dashboards
- **Overview dashboard:** total students, total tutors, exams scheduled today, exams completed, average score
- **Per-department dashboard:** student count, exam participation, average per subject
- **Users management page:** list/search students, view their registration details (name, email, department, subjects, student ID, exams taken, avg score), trigger password reset, edit profile fields, disable/enable account
- **Note on passwords:** Firebase does not expose passwords to anyone (they're hashed). Admin gets a "Reset password" button that sends the reset email or sets a temp password the admin can hand over.

### 9. Email verification
- Firebase sends verification email on signup
- Unverified users see a "verify your email" banner and cannot enter exams
- Password reset uses Firebase's flow with a dedicated `/reset-password` route

## Routes (TanStack Start file-based)
```
src/routes/
  __root.tsx
  index.tsx                       # landing / login redirect
  login.tsx
  register.tsx
  verify-email.tsx
  reset-password.tsx
  student/
    index.tsx                     # student dashboard (upcoming exams)
    exam.$sessionId.tsx           # exam taking UI
    exam.$sessionId.result.tsx    # score + corrections (when released)
  admin/
    index.tsx                     # admin overview dashboard
    questions.tsx                 # question bank manager
    exams.tsx                     # list + create exam sessions
    exams.$sessionId.tsx          # session detail + live scoreboard + release corrections
    users.tsx                     # user management (super_admin only)
    keys.tsx                      # product key generator (super_admin only)
    departments.tsx               # per-department analytics
```

Route guards via a `_authenticated` layout that checks Firebase user + role.
